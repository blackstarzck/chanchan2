# Deployment And Operations

This document explains how to maintain, version, and publish the `chanchan2` packages.

## Package Structure

The workspace currently publishes two packages:

- `@chanchan2/tokens`
- `@chanchan2/ui`

The docs app is internal:

- `@chanchan2/docs`

## Day-To-Day Workflow

Typical development flow:

1. Update Figma-backed token data if needed.
2. Run `npm run tokens:sync`.
3. Implement or update components in `packages/ui`.
4. Verify examples in `apps/docs`.
5. Run `npm run typecheck`.
6. Run `npm run build`.
7. Record a release note with `npm run changeset`.
8. Open or merge your pull request.

## Token Sync

Source of truth:

- [packages/tokens/src/token-source.json](C:\Users\chanki\workspace\creative-ui\chanchan2\packages\tokens\src\token-source.json)

Generated outputs:

- [packages/tokens/src/tokens.ts](C:\Users\chanki\workspace\creative-ui\chanchan2\packages\tokens\src\tokens.ts)
- [packages/tokens/theme.css](C:\Users\chanki\workspace\creative-ui\chanchan2\packages\tokens\theme.css)

Command:

```bash
npm run tokens:sync
```

What it does:

- reads the token source JSON
- regenerates TypeScript token exports
- regenerates the CSS variable contract

Important rule:

- never hand-edit the generated token files
- always edit `token-source.json` and re-run sync

## Changesets

Changesets records what should be released and how versions should change.

Command:

```bash
npm run changeset
```

What it asks:

1. which package changed
2. whether the change is `patch`, `minor`, or `major`
3. a short human-readable summary

What it creates:

- a markdown file in [\.changeset](C:\Users\chanki\workspace\creative-ui\chanchan2\.changeset)

That file is later consumed by the release workflow to update package versions and publish to npm.

## Release Automation

There are two workflows in this repo:

- CI: [\.github/workflows/ci.yml](C:\Users\chanki\workspace\creative-ui\chanchan2\.github\workflows\ci.yml)
- Release: [\.github/workflows/release.yml](C:\Users\chanki\workspace\creative-ui\chanchan2\.github\workflows\release.yml)

### CI workflow

The CI workflow runs on pull requests and on pushes to `main`.

It does the following:

1. installs dependencies
2. runs `npm run tokens:sync`
3. runs `npm run typecheck`
4. runs `npm run build`

This protects the repo from shipping broken packages.

### Release workflow

The release workflow runs on pushes to `main`.

It uses `changesets/action` to:

1. detect whether there are unpublished changesets
2. create or update a release PR if versions need to be bumped
3. publish packages to npm when the repo is ready to release

In simple terms:

- `npm run changeset` writes the release note
- merging to `main` triggers automation
- automation prepares versions and publishes

## NPM_TOKEN

The current release workflow uses `NPM_TOKEN`.

Important:

- `NPM_TOKEN` is not stored in this repository
- it is not generated by GitHub automatically
- you must create it in npm and add it to GitHub Actions secrets

### Where `NPM_TOKEN` comes from

You create it in your npm account:

1. sign in to [npmjs.com](https://www.npmjs.com/)
2. open your profile menu
3. go to `Access Tokens`
4. create a new token with `Read and write` access
5. copy the token immediately

The token value is usually shown only once right after creation.

### Where to put `NPM_TOKEN`

You store it in GitHub repository secrets:

1. open your GitHub repository
2. go to `Settings`
3. open `Secrets and variables`
4. open `Actions`
5. create a new repository secret
6. name it `NPM_TOKEN`
7. paste the token value from npm

So if you ask "where is `NPM_TOKEN`?" the answer is:

- it should live in GitHub Actions secrets
- its original source is your npm account access token page

## GITHUB_TOKEN vs NPM_TOKEN

These are different:

- `GITHUB_TOKEN`
  - automatically provided by GitHub Actions
  - used to create release PRs and commit workflow changes
- `NPM_TOKEN`
  - not automatic
  - must be created by you in npm
  - used to authenticate `npm publish`

## First-Time Setup Checklist

Before the first public release:

1. create the npm organization or confirm the scope name
2. make sure package names are correct in each `package.json`
3. create the `NPM_TOKEN` in npm
4. save `NPM_TOKEN` in GitHub Actions secrets
5. make sure the default branch is `main`
6. push the repository with the workflow files
7. run `npm run changeset`
8. merge to `main`
9. inspect the created release PR or publish run in GitHub Actions

## Manual Commands

Useful local commands:

```bash
npm run tokens:sync
npm run typecheck
npm run build
npm run changeset
npm run version-packages
```

If you ever need a local manual publish:

```bash
npm publish --workspace @chanchan2/tokens --access public
npm publish --workspace @chanchan2/ui --access public
```

Normally, CI should handle publish once `main` is configured correctly.

## When To Use Patch, Minor, Major

Use `patch` when:

- fixing styles
- fixing behavior without breaking APIs
- improving docs

Use `minor` when:

- adding new components
- adding new non-breaking props or variants
- extending tokens without breaking consumers

Use `major` when:

- renaming exports
- removing props or components
- changing token names that consumers rely on
- making styling contracts incompatible

## Recommended Management Rules

To keep the package healthy:

- treat `token-source.json` as the only editable token source
- do not merge component changes without docs examples
- always add a changeset for publishable package changes
- do not publish directly from a local machine unless you are recovering from CI failure
- keep component APIs small and token-driven
- prefer semantic token names over brand-specific names inside the public component layer

## Recommended Future Improvement

The current workflow uses `NPM_TOKEN` because it is easy to understand and works immediately.

A stronger long-term option is npm trusted publishing with GitHub Actions OIDC.

Benefits:

- no long-lived npm token stored in GitHub
- less secret management
- better security posture
